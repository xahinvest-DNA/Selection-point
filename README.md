# Selection Point

**Selection Point** — самостоятельная система восстановления способности выбора в ситуациях внутреннего и внешнего напряжения.

Проект развивается не через раннее создание курса или приложения, а через последовательное утверждение методологической архитектуры.

## Текущее состояние

- Активная фаза: **Фаза 3 — точная архитектура пяти ступеней**.
- Активная ступень: **Ступень 1 «Жизнь внутри автоматизма»**.
- Утверждены параметры 1–4:
  1. состояние человека;
  2. способ восприятия реальности;
  3. способ восприятия себя;
  4. что управляет решениями.
- Следующий вопрос: **SP-S1-P05 — доступная точка выбора**.

Актуальная контрольная точка находится в:

- `docs/FOUNDATION/CURRENT_PROJECT_STATE.md`;
- `docs/CURRENT_STATE.md` — короткая совместимая точка входа.

## Что создаётся

Selection Point помогает человеку:

1. замечать автоматическую реакцию;
2. отличать факт от собственной модели ситуации;
3. возвращать доступную часть выбора;
4. формировать внутреннюю задачу;
5. совершать точный следующий ход;
6. использовать последствия как обратную связь;
7. постепенно изменять систему условий, поддерживающих повторяющееся поведение.

Первое прикладное поле — тело и повседневная жизнь: питание, сон, движение, восстановление, энергия, пищевые и другие импульсы, алкоголь как способ снятия напряжения, бытовое избегание и возвращение после срыва.

Тело является первым полем практики, но не окончательной границей метода.

## Архитектура знаний

Чтобы сохранять глубину и не превращать будущий курс в массив теоретических файлов, проект разделён на четыре слоя:

```text
полная теория
→ каноническая архитектура
→ общие библиотеки
→ материалы доставки
```

### Полная теория

Папки `*_THEORY/` сохраняют весь ход разбора, аргументацию, примеры, исключения, риски и утверждённую формулировку. Эти файлы не сокращаются ради будущего курса.

### Каноническая архитектура

`docs/FOUNDATION/CANONICAL/` хранит компактное действующее представление утверждённой системы:

- восемь положений модели человека и изменения;
- канонические карточки параметров ступеней;
- правила дельты;
- шаблон карточки;
- правила будущего слоя доставки.

### Общие библиотеки

`docs/FOUNDATION/LIBRARIES/` хранит:

- стабильные ID терминов;
- реестр утверждений;
- переиспользуемые примеры;
- общие риски и границы;
- доказательный статус;
- перекрёстные связи.

### Материалы доставки

Курс, практики, рабочая тетрадь, пилот и приложение создаются позже из канонической архитектуры и утверждённых практик. Они не создаются прямым сокращением полного теоретического файла.

## Источники истины

Работу необходимо начинать с документов в следующем порядке:

1. `docs/FOUNDATION/PROJECT_OPERATING_PROTOCOL.md`;
2. `docs/FOUNDATION/CURRENT_PROJECT_STATE.md`;
3. `docs/FOUNDATION/ARCHITECTURAL_DECLARATION.md`;
4. `docs/FOUNDATION/HUMAN_CHANGE_MODEL.md`;
5. `docs/FOUNDATION/SOURCE_SYNTHESIS_PRINCIPLE.md`;
6. `docs/FOUNDATION/CANONICAL/00_INDEX.md`;
7. `docs/FOUNDATION/CANONICAL/00_CANONICAL_RULES.md`;
8. `docs/FOUNDATION/LIBRARIES/00_INDEX.md`;
9. `docs/FOUNDATION/FIVE_STAGES_THEORY/00_INDEX.md`;
10. полный и канонический файлы активной ступени.

При расхождении старых документов с `docs/FOUNDATION/` приоритет всегда имеет утверждённая архитектура в `docs/FOUNDATION/`.

Канонический слой является производным. Он не может изменять смысл полного утверждённого теоретического источника.

## Рабочий принцип

```text
один вопрос
→ полный теоретический разбор
→ обсуждение
→ явное утверждение Андреем
→ фиксация полного материала
→ создание канонической карточки
→ обновление библиотек
→ проверка файлов
→ обновление контрольной точки
→ следующий вопрос
```

Нельзя переходить к следующему параметру, модулю, практике или фазе до выполнения всего цикла.

Начиная с SP-S1-P05 действует правило дельты: новый теоретический файл подробно раскрывает только новый вопрос, а прежние положения обозначает ID и ссылками.

## Что сейчас не утверждено

На текущей фазе не утверждены:

- количество модулей и уроков;
- длительность первой версии;
- 14-дневный формат;
- формат курса;
- мобильное приложение;
- AI-сопровождение;
- Expo, React Native или другой технический стек;
- backend, база данных, авторизация и платежи.

Эти решения принимаются только на последующих фазах после завершения архитектуры метода и проверки ручного пилота.

## Структура репозитория

- `docs/FOUNDATION/` — действующая утверждённая архитектура и текущая контрольная точка.
- `docs/FOUNDATION/CANONICAL/` — компактная рабочая архитектура утверждённых положений.
- `docs/FOUNDATION/LIBRARIES/` — переиспользуемые термины, тезисы, примеры, риски и связи.
- `docs/CONTENT/` — ранние материалы и продуктовые гипотезы; не являются источником истины.
- `docs/METHOD_CORE.md`, `docs/PRACTICE_SYSTEM.md`, `docs/PRODUCT_BRIEF.md`, `docs/DECISIONS.md` — прежние черновики и исторические решения.
- `docs/LEGACY_STATUS.md` — реестр устаревших материалов.
- `codex/` — исторические или будущие исполнительские задания. Ни одно задание не считается активным без прямой ссылки из актуальной контрольной точки.

## Роли

- **Андрей** — владелец метода и продукта, утверждает каждое ключевое положение.
- **ChatGPT** — методологический архитектор и ведущий проекта: удерживает целостность, раскрывает один вопрос, фиксирует полный и канонический слои, обновляет библиотеки и контрольную точку.
- **Codex** — исполнитель только уже утверждённых и измеримых задач. Codex не определяет философию, смысл ступеней, содержание курса или формат продукта.

Главное правило проекта:

> Не скорость генерации, а последовательность утверждённых решений создаёт архитектуру Selection Point.
