# Канонический слой Selection Point

**Статус:** утверждённый обязательный регламент  
**Дата введения:** 4 июля 2026 года  
**Дата обновления:** 5 июля 2026 года  
**Основание:** решения Андрея о разделении полного теоретического материала, рабочей архитектуры и сквозных визуальных моделей

## 1. Назначение

Канонический слой сохраняет утверждённый смысл в компактной рабочей форме, не перенося весь объём теоретических файлов непосредственно в курс, практики, пилот или цифровой продукт.

Проект использует четыре основных слоя:

```text
полная теория
→ каноническая архитектура
→ общие библиотеки
→ материалы доставки
```

### Слой 1. Полная теория

Папки `*_THEORY/` и полные файлы параметров содержат:

- полный ход рассуждения;
- аргументацию;
- примеры;
- различения;
- альтернативные трактовки;
- риски и ограничения;
- утверждённую итоговую формулировку;
- архитектурные следствия.

Полные теоретические файлы не сокращаются ради удобства будущего курса и не переписываются молча после утверждения.

### Слой 2. Каноническая архитектура

Папка `docs/FOUNDATION/CANONICAL/` содержит компактное действующее представление утверждённой системы.

Канонический слой:

- фиксирует только утверждённое;
- сохраняет смысл полного материала;
- показывает, что добавляет каждое положение;
- устраняет повторное объяснение уже принятой основы;
- является основным входом для следующих фаз;
- всегда содержит ссылки на полный источник.

Канонический файл не заменяет полный теоретический файл.

### Слой 3. Общие библиотеки

Папка `docs/FOUNDATION/LIBRARIES/` содержит:

- термины;
- утверждения и стабильные ID;
- примеры;
- риски и границы;
- доказательный статус;
- перекрёстные связи.

Общие элементы не переписываются подробно в каждом новом параметре.

### Слой 4. Материалы доставки

Курс, рабочая тетрадь, практики, заметки преподавателя, пилот и цифровой интерфейс создаются только на соответствующих фазах.

Они собираются из:

```text
каноническая архитектура
+ утверждённая практическая архитектура
+ общие библиотеки
+ критерии освоения
+ утверждённый формат
```

Полная теория используется для проверки смысла, но не переносится в урок автоматически.

---

## 2. Сквозной слой визуальных моделей

`docs/FOUNDATION/VISUAL_MODELS/` является вспомогательным слоем представления, а не пятым основным слоем и не самостоятельным источником истины.

Визуальная модель:

- создаётся только из утверждённых положений;
- получает стабильный ID вида `SP-VM-01`;
- связывает несколько канонических тезисов;
- содержит границы метафоры;
- связывается с терминами, примерами, рисками и доказательным статусом;
- может использоваться в будущем слое доставки только после проверки её учебной функции;
- не должна незаметно добавлять новый смысл.

Если визуальная модель противоречит полному источнику, приоритет имеет полный утверждённый теоретический файл.

---

## 3. Иерархия при расхождении

При расхождении применяется следующий порядок:

1. явно утверждённое решение Андрея;
2. фундаментальные документы `docs/FOUNDATION/`;
3. полный утверждённый теоретический файл;
4. канонический файл;
5. библиотеки и визуальные модели;
6. будущие материалы доставки;
7. legacy-материалы.

Если каноническая карточка или визуальная модель расходится с полным источником, работа останавливается. Расхождение показывается Андрею и устраняется только после решения.

---

## 4. Стабильные идентификаторы

Формат:

```text
SP-HCM-01       положение модели человека
SP-S1-P01       ступень 1, параметр 1
SP-S2-P01       ступень 2, параметр 1
SP-TERM-STATE   термин
SP-EX-FOOD-001  пример
SP-RISK-001     риск или граница
SP-VM-01        визуальная модель
```

ID не меняется при редакционном уточнении названия. Существенное изменение смысла фиксируется в истории и требует отдельного решения.

---

## 5. Обязательная структура канонической карточки

Для каждого параметра фиксируются:

1. ID;
2. название;
3. статус;
4. полный источник;
5. центральный тезис;
6. что именно добавляет параметр;
7. механизм;
8. наблюдаемые признаки;
9. главные различения;
10. границы и ограничения;
11. связи с утверждениями, терминами, примерами и рисками;
12. связанные визуальные модели, если они существуют;
13. архитектурные следствия;
14. утверждённая формулировка в компактном виде;
15. короткая формула.

Каноническая карточка должна быть достаточной для проектирования следующего слоя, но не обязана сохранять весь путь рассуждения.

Ориентир объёма завершённой карточки — примерно 500–900 слов. Это ориентир, а не механический лимит.

---

## 6. Правило дельты

Каждый новый полный теоретический файл подробно раскрывает только то, что добавляет к утверждённой системе.

В начале файла указываются:

- утверждённые ID, на которые он опирается;
- один новый вопрос;
- положения, которые не пересматриваются;
- соседние параметры, которые намеренно не рассматриваются.

Уже утверждённая теория:

- кратко напоминается;
- обозначается стабильными ID;
- связывается ссылками;
- не пересказывается заново без необходимости.

Допустимое повторение:

- необходимо для понимания нового перехода;
- фиксирует границу вопроса;
- создаёт новый вывод при соединении прежних положений.

Недопустимое повторение:

- заново объясняет весь предыдущий параметр;
- повторяет общие ограничения вместо ссылки на реестр;
- накапливает однотипные примеры;
- скрыто переписывает утверждённый смысл;
- использует новую схему как замену аргументации.

---

## 7. Рабочий цикл после утверждения параметра

После явного утверждения ChatGPT обязан в том же рабочем цикле:

1. сохранить полный теоретический файл;
2. проверить его чтением из репозитория;
3. создать или обновить каноническую карточку;
4. обновить `CLAIMS_REGISTRY.md`;
5. добавить только новые термины в `TERMS_REGISTRY.md`;
6. вынести повторно используемые примеры в `EXAMPLE_LIBRARY.md`;
7. добавить новые общие ограничения в `RISKS_AND_BOUNDARIES.md`;
8. обновить `CROSS_REFERENCE_MAP.md`;
9. обновить `EVIDENCE_MAP.md`;
10. создать или обновить визуальную модель, если она утверждена;
11. проверить границы визуальной метафоры;
12. проверить соблюдение правила дельты;
13. обновить индекс активной фазы;
14. обновить `CURRENT_PROJECT_STATE.md` и операционный протокол;
15. только затем перейти к следующему вопросу.

## 8. Статусы канонических карточек и моделей

- `не рассмотрено` — тема ещё не обсуждалась;
- `на обсуждении` — разбор начат, но не утверждён;
- `утверждено` — существует утверждённый источник и производная форма;
- `требует пересмотра` — обнаружено значимое противоречие;
- `архив` — материал больше не управляет архитектурой.

Черновой текст или набросок нельзя помещать в канонический или визуальный слой как утверждённый.

---

## 9. Правило использования в будущем курсе

Будущий урок не создаётся прямым сокращением большого теоретического файла.

Рабочая последовательность:

```text
канонический тезис
→ конкретная учебная задача
→ минимально необходимая теория
→ подходящий пример
→ визуальная модель, если она улучшает различение
→ утверждённая практика
→ наблюдаемое действие
→ обратная связь
→ критерий освоения
```

Визуальная модель включается только если:

- помогает выполнить учебную задачу;
- не создаёт ложной простоты или иллюзии полного контроля;
- понятна без длинного устного исправления;
- её ограничения явно обозначены;
- понимание проверено в пилоте.

Полный теоретический файл остаётся методологическим основанием и расширенным материалом для автора или преподавателя.

---

## 10. Главное правило

> Полнота сохраняется в теоретическом слое. Управляемость создаётся в каноническом слое. Повторяемые элементы живут в библиотеках. Визуальные модели показывают связи, не заменяя смысл. Курс и продукт собираются только из утверждённой архитектуры.
