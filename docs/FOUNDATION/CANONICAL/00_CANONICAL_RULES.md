# Канонический слой Selection Point

**Статус:** утверждённый обязательный регламент  
**Дата введения:** 4 июля 2026 года  
**Основание:** решение Андрея о разделении полного теоретического материала и рабочей архитектуры

## 1. Назначение

Канонический слой создаётся для того, чтобы сохранять полную глубину исследования, но не переносить весь объём теоретических файлов непосредственно в курс, практики, пилот или цифровой продукт.

Проект использует четыре разных слоя:

```text
полная теория
→ каноническая архитектура
→ общие библиотеки
→ материалы доставки
```

### Слой 1. Полная теория

Папки `*_THEORY/` содержат:

- полный ход рассуждения;
- аргументацию;
- примеры;
- различения;
- альтернативные трактовки;
- риски и ограничения;
- утверждённую итоговую формулировку;
- архитектурные следствия.

Полные теоретические файлы не сокращаются ради удобства будущего курса и не переписываются молча после утверждения.

### Слой 2. Каноническая архитектура

Папка `docs/FOUNDATION/CANONICAL/` содержит компактное действующее представление утверждённой системы.

Канонический слой:

- фиксирует только утверждённое;
- сохраняет смысл полного материала;
- показывает, что именно добавляет каждое положение;
- устраняет повторное объяснение уже принятой основы;
- является основным входом для проектирования курса, практик и продукта;
- всегда содержит ссылки на полный теоретический источник.

Канонический файл не заменяет и не отменяет полный теоретический файл.

### Слой 3. Общие библиотеки

Папка `docs/FOUNDATION/LIBRARIES/` содержит переиспользуемые элементы:

- термины;
- утверждения и их стабильные идентификаторы;
- примеры;
- риски и границы;
- карту внешней доказательной проверки;
- перекрёстные связи.

Общие элементы не должны заново подробно переписываться в каждом параметре.

### Слой 4. Материалы доставки

Курс, рабочая тетрадь, практики, заметки преподавателя, пилот и цифровой интерфейс создаются только на соответствующих фазах.

Они собираются из:

```text
каноническая архитектура
+ утверждённая практическая архитектура
+ общие библиотеки
```

Полная теория используется для проверки смысла, разрешения противоречий и подготовки расширенного объяснения, но не переносится в урок автоматически.

---

## 2. Иерархия при расхождении

При расхождении применяется следующий порядок:

1. явно утверждённое решение Андрея;
2. фундаментальные документы `docs/FOUNDATION/`;
3. полный утверждённый теоретический файл;
4. канонический файл;
5. библиотеки;
6. будущие материалы доставки;
7. legacy-материалы.

Если каноническая карточка и полный теоретический файл расходятся по смыслу, работа останавливается. Расхождение показывается Андрею и устраняется только после решения. Нельзя молча исправлять утверждённый смысл.

---

## 3. Стабильные идентификаторы

Каждое утверждённое положение получает постоянный ID.

Формат:

```text
SP-HCM-01       положение модели человека
SP-S1-P01       ступень 1, параметр 1
SP-S2-P01       ступень 2, параметр 1
SP-TERM-STATE   термин
SP-EX-FOOD-001  пример
SP-RISK-001     риск или граница
```

ID не меняется при редакционном уточнении названия. Если смысл положения изменён существенно, изменение фиксируется в истории и при необходимости создаётся новая версия карточки.

---

## 4. Обязательная структура канонической карточки

Для каждого параметра ступени фиксируются:

1. ID;
2. название;
3. статус;
4. полный источник;
5. центральный тезис;
6. что именно добавляет параметр;
7. механизм;
8. наблюдаемые признаки;
9. главные различения;
10. границы и ограничения;
11. связи с другими утверждениями;
12. архитектурные следствия;
13. утверждённая формулировка в компактном виде;
14. короткая формула.

Каноническая карточка должна быть достаточной для проектирования следующего слоя, но не обязана сохранять все примеры и весь путь рассуждения.

Ориентир объёма одной завершённой карточки — примерно 500–900 слов. Это ориентир, а не механический лимит.

---

## 5. Правило дельты

Каждый новый полный теоретический файл подробно раскрывает только то, что он добавляет к уже утверждённой системе.

В начале нового файла необходимо указать:

- на какие утверждённые положения он опирается;
- какой один новый вопрос рассматривается;
- какие соседние параметры намеренно не рассматриваются.

Уже утверждённая теория:

- кратко напоминается;
- обозначается стабильными ID;
- связывается ссылками;
- не пересказывается заново на десятках страниц без отдельной необходимости.

Допустимое повторение:

- краткое напоминание, необходимое для понимания;
- точное различение границы текущего вопроса;
- новый вывод, который появляется только при соединении прежних положений.

Недопустимое повторение:

- повтор полного объяснения предыдущего параметра;
- повтор одинаковых общих ограничений вместо ссылки на реестр;
- накопление множества однотипных примеров внутри каждого файла;
- использование нового документа как скрытого переписывания уже утверждённого смысла.

---

## 6. Рабочий цикл после утверждения параметра

После явного утверждения каждого нового параметра ChatGPT обязан в том же рабочем цикле:

1. сохранить полный теоретический файл;
2. проверить его чтением из репозитория;
3. создать или обновить каноническую карточку параметра;
4. обновить `CLAIMS_REGISTRY.md`;
5. добавить только новые термины в `TERMS_REGISTRY.md`;
6. вынести новые повторно используемые примеры в `EXAMPLE_LIBRARY.md`;
7. добавить новые общие ограничения в `RISKS_AND_BOUNDARIES.md`;
8. обновить `CROSS_REFERENCE_MAP.md`;
9. отметить статус внешней доказательной проверки в `EVIDENCE_MAP.md`;
10. проверить соблюдение правила дельты;
11. обновить индекс активной фазы;
12. обновить `CURRENT_PROJECT_STATE.md` и контрольную точку операционного протокола;
13. только затем перейти к следующему вопросу.

---

## 7. Статусы канонических карточек

- `не рассмотрено` — параметр ещё не обсуждался;
- `на обсуждении` — разбор начат, но не утверждён;
- `утверждено` — существует полный источник и каноническая карточка;
- `требует пересмотра` — обнаружено значимое противоречие;
- `архив` — карточка больше не управляет архитектурой.

Черновой или обсуждаемый текст нельзя помещать в канонический слой как утверждённый.

---

## 8. Правило использования в будущем курсе

Будущий урок не должен создаваться прямым сокращением большого теоретического файла.

Рабочая последовательность:

```text
канонический тезис
→ конкретная учебная задача
→ минимально необходимая теория
→ один или несколько подходящих примеров
→ утверждённая практика
→ наблюдаемое действие
→ обратная связь
→ критерий освоения
```

Полный теоретический файл остаётся методологическим основанием и расширенным материалом для автора или преподавателя.

---

## 9. Главное правило

> Полнота сохраняется в теоретическом слое. Управляемость создаётся в каноническом слое. Повторяемые элементы живут в библиотеках. Курс и продукт собираются только из утверждённой архитектуры.
